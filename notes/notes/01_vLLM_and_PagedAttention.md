# vLLM 与 PagedAttention

## 1. 什么是 vLLM？

vLLM 是一个面向 **Transformer 架构大语言模型** 的高性能推理引擎。  
它的核心目标很明确：**在有限显存条件下，提高推理吞吐量，并尽可能提升显存利用率。**

对于需要部署大模型推理服务的场景来说，vLLM 的价值主要体现在两点：

- 更高的并发处理能力
- 更高效的 KV Cache 管理方式

而这其中，最关键的技术之一就是 **PagedAttention**。

---

## 2. 什么是 PagedAttention？

PagedAttention 是 vLLM 的核心技术之一。  
如果要用一句话概括它的思想，可以理解为：

> **将原本要求连续存储的 KV Cache，拆分成固定大小的块，分散存放，再通过映射关系统一管理。**

它主要解决的是 **KV Cache 显存管理效率低** 的问题。

---

## 3. PagedAttention 解决了什么问题？

在大语言模型生成文本时，系统需要持续保存每个 Token 对应的 **Key-Value Cache（KV Cache）**。  
传统方法通常要求这部分缓存占用一段 **连续的显存空间**，这会带来两个明显问题：

1. **显存碎片化严重**  
   随着请求不断进入、结束、释放，显存中会出现大量不连续的小空洞，难以继续高效利用。

2. **显存浪费明显**  
   为了避免运行过程中频繁扩容，传统方案往往会为请求预留较大空间，但很多请求实际并不会用满，导致大量显存被白白占用。

PagedAttention 的思路则更像操作系统中的分页机制：  
它不再强求 KV Cache 连续存放，而是把它拆分为固定大小的块进行管理，从而显著减少显存碎片和浪费。

---

## 4. PagedAttention 的核心原理

### 4.1 逻辑块（Logical Blocks）

可以把一个请求生成过程中的 KV Cache 看作一串逻辑上连续的数据。  
PagedAttention 会先把这串数据切分成多个固定大小的 **逻辑块**。

例如，假设每个块能容纳 4 个 Token 的 KV 信息，那么一个长度为 7 的序列就会被划分为：

- Block 0：前 4 个 Token
- Block 1：后 3 个 Token（剩余 1 个位置暂时空闲）

---

### 4.2 物理块（Physical Blocks）

在 GPU 显存中，vLLM 会维护许多固定大小的 **物理块**。  
这些物理块在显存中的位置可以是**不连续的**。

也就是说，一个请求的逻辑块不需要顺序地挨在一起存放，只要有可用空间，就可以分配到任意空闲物理块中。

---

### 4.3 块表（Block Table）

既然逻辑块和物理块不再一一连续对应，就需要一个映射结构进行管理，这就是 **Block Table**。

它负责记录类似这样的关系：

- 逻辑上的第 0 块，对应物理上的第 12 块
- 逻辑上的第 1 块，对应物理上的第 27 块

这样一来，虽然数据在物理上是离散存储的，但在逻辑上依然可以被当作连续序列来访问。

---

## 5. PagedAttention 的工作流程

### 5.1 分块（Partitioning）

当一个 Prompt 进入系统后，vLLM 会先把对应的 Token 序列切分成固定大小的块。

例如：

- Block Size = 4
- 输入共有 7 个 Token

那么就会被切分为两个逻辑块：

- Block 0：前 4 个 Token
- Block 1：后 3 个 Token

---

### 5.2 动态分配（Dynamic Allocation）

在生成过程中，vLLM 不会一开始就为整个请求预留大量连续显存。  
相反，它采取的是 **按需分配** 的方式：

- 当前块未写满时，继续写入当前块
- 当前块写满后，再去显存池申请新的物理块

这个新的物理块可以位于显存中的任意位置，并不要求和前一个块相邻。

这种机制显著减少了显存预留带来的浪费。

---

### 5.3 高效读取（Kernel Execution）

这是 PagedAttention 最关键、也最难实现的一部分。

在进行 Attention 计算时，GPU 并不会因为 KV Cache 被分散存储就“找不到数据”。  
vLLM 为此专门实现了优化过的 CUDA Kernel，其执行逻辑大致如下：

1. 根据 Block Table 定位当前请求对应的所有物理块
2. 从这些分散的物理块中高效读取所需的 KV 数据
3. 并行完成 Attention 计算

也就是说，虽然底层存储是离散的，但通过专门设计的内核，读取和计算依然可以保持较高效率。

---

## 6. PagedAttention 的优势

### 6.1 更高的吞吐量

由于显存管理更高效、调度更灵活，vLLM 在相同硬件条件下通常能够支持更大的 batch 和更高的并发请求数，从而带来更高的整体吞吐量。

---

### 6.2 更高的显存利用率

传统方案中，大量显存会浪费在：

- 连续空间预留
- 显存碎片
- 无法复用的空洞区域

PagedAttention 通过块化管理和映射机制，大幅提升了显存利用率，使得同一张 GPU 可以同时承载更多请求。

---

### 6.3 更低的排队延迟

vLLM 结合 **Continuous Batching（持续批处理）** 技术，可以让新请求在合适时机快速加入当前运行中的批次，而不需要等上一整个 batch 全部结束。

这能有效缓解传统静态批处理中常见的“长尾等待”问题。

---

## 7. vLLM 的典型使用场景

### 7.1 高并发 API 服务

如果一个应用需要同时为大量用户提供对话或文本生成服务，那么 vLLM 会是非常合适的后端选择。

这也是我在做智能体后端推理服务时，优先考虑 vLLM 的主要原因之一。

---

### 7.2 成本优化

通过提高单卡吞吐量和显存利用率，vLLM 往往能够减少所需 GPU 数量，从而降低部署成本。

---

### 7.3 私有化部署

在企业内部部署开源模型时，vLLM 可以作为统一的推理层，对外提供标准化接口。  
例如部署 Qwen、DeepSeek 等模型时，vLLM 都是一个很常见的选项。

---

### 7.4 长上下文生成

由于它对 KV Cache 的管理更高效，因此在长文本、长上下文推理场景下，vLLM 往往比传统方案表现更稳健。

---

## 8. vLLM 的运行流程

下面从整体角度简要梳理一下 vLLM 的推理流程。

### 8.1 请求接入与封装

当用户通过 API（例如 OpenAI 兼容接口）发送 Prompt 后，系统会先完成以下步骤：

#### 8.1.1 Tokenization（分词）

首先将输入文本转换为模型可以处理的 Token ID 序列。

#### 8.1.2 请求封装

每个请求会被封装为一个 **SequenceGroup**。  
如果场景涉及多轮对话、Beam Search 等，一个组中可能包含多个序列。

#### 8.1.3 进入等待队列

请求不会立即执行，而是先进入等待队列，等待调度器分配计算资源。

---

### 8.2 核心调度与持续批处理

这是 vLLM 性能优势非常重要的一部分。

#### 8.2.1 静态批处理的问题

传统推理框架通常采用 **静态批处理**：

- 一个 batch 内的请求一起开始
- 必须等这个 batch 中所有请求都结束后，才能处理下一批

这样会带来明显的长尾效应：  
如果其中某个请求生成得特别长，其他较短请求也必须跟着等待。

#### 8.2.2 Continuous Batching 的改进

vLLM 采用的是 **迭代级调度** 思路：

- 每完成一轮 Token 生成，调度器都会重新检查当前状态
- 已完成的请求会被及时移出 batch
- 新请求只要资源允许，就可以尽快加入当前 batch

因此，GPU 不需要等整批请求全部结束后再继续工作，资源利用率会更高。

#### 8.2.3 调度策略

调度器会结合当前显存使用情况，尤其是可用物理块数量，决定哪些请求可以从等待队列进入运行队列。

---

### 8.3 显存管理

当请求进入运行阶段后，Block Manager 会参与 KV Cache 的管理。

#### 8.3.1 逻辑块与物理块映射

系统会在显存池中分配若干不连续的物理块来存储 KV Cache，  
并维护一张映射表，记录逻辑序列与物理块之间的对应关系。

#### 8.3.2 动态按需扩展

在生成初期，只会分配当前真正需要的块。  
随着生成 Token 数量增多，如果已有块写满，Block Manager 才会继续申请新的物理块。

这种方式避免了“大量预留、实际闲置”的问题。

---

### 8.4 模型执行

从执行视角来看，推理通常可以分为两个阶段。

#### 阶段 A：Prefill（预填充）

这一阶段主要处理用户输入的整个 Prompt。

特点是：

- 需要一次性处理较多 Token
- 计算密集
- 会并行生成 Prompt 对应的 KV Cache

#### 阶段 B：Decoding（解码）

这一阶段主要负责逐个生成新 Token。

特点是：

- 每次只生成少量 Token
- 更依赖显存读写效率
- 会频繁调用 PagedAttention 内核读取分散存储的 KV Cache

---

### 8.5 输出处理与请求结束

#### 8.5.1 采样（Sampling）

模型输出概率分布后，系统会基于设定策略选择下一个 Token，例如：

- Greedy
- Top-K
- Top-P

#### 8.5.2 流式返回（Streaming）

生成出的 Token 可以实时返回给用户，从而提升交互体验。

#### 8.5.3 显存回收

当请求结束后：

- 调度器会将其从运行队列中移除
- 对应的物理块会被标记为空闲
- 显存资源重新回到显存池中，供后续请求复用

---

## 9. 我的理解

vLLM 的真正价值是它通过 **PagedAttention + Continuous Batching**，重新设计了大模型推理服务中的两个核心问题：

- **如何更高效地管理 KV Cache**
- **如何让 GPU 在高并发场景下尽量不被浪费**

也正因如此，vLLM 不只是一个“能跑模型”的推理框架，更像是一个专门为大模型在线服务优化过的推理系统。

---

## 10. 总结

> **vLLM 通过 PagedAttention 改进了 KV Cache 的管理方式，通过 Continuous Batching 改进了请求调度方式，从而提升了大模型推理服务中的吞吐量、显存利用率和整体工程效率。**
